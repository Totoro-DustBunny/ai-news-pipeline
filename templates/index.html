What This Pipeline Does

This pipeline automatically monitors 12 trusted AI news sources — including venture capital blogs, AI company announcements, and leading technology publications — collecting all articles published within the last 7 days. New articles are fetched, deduplicated, and stored in a local database each time the pipeline runs.

Each article is evaluated by a large language model via OpenRouter, which assigns a relevance score from 1 to 10 based on how closely the article relates to AI tools, SaaS products, and practical AI adoption. Articles scoring 7 or above are retained and classified into one of five business categories — surfacing what matters for professionals and decision-makers without manual curation.

LinkedIn Post Anatomy — Writing Checklist

Use this checklist when drafting LinkedIn posts inspired by top KOL patterns.

🔎 Hook (First Line)

  • Opens with a bold claim, surprising stat, or rhetorical question
  • Creates immediate curiosity or challenges a common assumption
  • Short enough to read before the "see more" cut-off (~200 chars)

📝 Structure & Formatting

  • Short paragraphs (1–3 lines) with deliberate line breaks
  • Numbered lists or bullet points for scannable content
  • Emojis used sparingly for visual anchoring, not decoration

🏆 Credibility Signals

  • Cites specific tools, companies, or data points by name
  • References a personal experience or domain-specific insight
  • Avoids vague generalities — every claim is grounded

💬 Engagement Closer

  • Ends with a direct question to the audience
  • Includes a CTA: tag someone, share, or comment with their view
  • Invites discussion rather than just broadcasting information

Flask and SQLite were selected deliberately over heavier alternatives like FastAPI + PostgreSQL or a cloud database. For a single-user analytical pipeline running on a local machine, SQLite's zero-configuration setup, file-based persistence, and native Python integration eliminate infrastructure overhead entirely. Flask's minimal footprint lets the project stay focused on the AI pipeline logic rather than deployment scaffolding — a conscious trade-off that prioritises iteration speed over production scalability.

The two-stage LLM pipeline — a relevance router followed by a classifier — separates two fundamentally different tasks. The router asks a lightweight question: is this article relevant to AI tools and SaaS? The classifier asks a harder question: which of five nuanced categories does it belong to? Splitting these concerns means the cheap Llama 3.3 70B model handles the bulk of the work (122 articles), while the stronger Llama 4 Maverick is only invoked for the 61 articles that passed the relevance gate — cutting token spend by approximately 50% compared to running a single combined prompt on the full article set.

OpenRouter's fallback model logic adds a resilience layer that would otherwise require custom retry logic. By declaring a primary model and a named fallback (mistral-nemo) in code, any transient API failure, quota limit, or model degradation is handled automatically without crashing the pipeline. In production, Llama 3.3 70B handled all 122 articles without triggering a single fallback — but the safety net was present from the first run, which is the right engineering discipline.

ArXiv Volume Overflow

ProblemArXiv CS.AI produced 334 articles in a single day — overwhelming the pipeline and making LLM scoring cost-prohibitive.
ResolutionRemoved ArXiv from sources.yaml entirely and replaced with the more curated AI Business feed.
2

Date Filtering Not Implemented at Launch

ProblemInitial ingest.py had no date filter, pulling full historical archives (1,482 articles including 908 from OpenAI alone).
ResolutionAdded a 7-day rolling window filter using feedparser's entry.published_parsed field.
3

API Cost Management

ProblemRunning classification prompts through premium models on 122 articles risked excessive token spend.
ResolutionSplit the pipeline into a cheap router (Llama 3.3 70B) for bulk filtering, then a stronger classifier (Llama 4 Maverick) only for the 61 relevant articles.
4

HBR SSL Certificate Failure

ProblemHarvard Business Review's RSS feed rejected connections due to an SSL verification error.
ResolutionLogged the error gracefully and continued ingestion — HBR remains in sources.yaml as enabled for future retry.

LinkedIn Data Access

ProblemLinkedIn has no public API and scraping violates their ToS — KOL post data cannot be retrieved programmatically with full fidelity.
ResolutionUsed DuckDuckGo web search to surface publicly quoted LinkedIn posts and combined with hardcoded KOL analysis based on known content patterns.

🔀

Separate Routing from Classifying

Running a cheap, fast model as a relevance gate before a more capable classifier is significantly more cost-efficient than using one powerful model for everything. The two-stage approach cut token usage by ~50% compared to running Llama 4 Maverick on all 122 articles.

📐

Structured Output is Non-Negotiable

LLM pipelines that depend on JSON parsing require explicit, unambiguous output format instructions. A single vague prompt cost more in debugging time than rewriting the prompt cleanly.

🌐

Source Curation Matters More Than Volume

More sources ≠ better signal. ArXiv's volume made it unusable without further filtering. Curated, high-signal sources (VC blogs, AI company blogs, industry news) outperformed academic feeds for business-relevant insight.

🛡️

Build Resilience From Day One

Adding fallback models and graceful error handling in the first version — rather than as an afterthought — meant zero pipeline crashes in production. Every external API call should anticipate failure.
